Section 1Approaches considered

The problem, precisely

CLIP's dual encoder trains on whole-image and whole-caption contrastive pairs. There is no mechanism in that objective for binding an attribute to a specific object, so it collapses into a bag of words: an image of a red tie and a white shirt produces an embedding nearly identical to an image of a white tie and a red shirt, because both captions contain the same words in a different arrangement. This is the exact failure the assignment names, and it is the fifth evaluation query by construction. Fine-tuned fashion embedding models (Marqo-FashionCLIP, Marqo-FashionSigLIP) close the vocabulary gap, better garment naming, better color naming, but not the binding gap, since they still produce one pooled vector per image.

Location (where someone is) and style (the vibe of an outfit) are scene-level signals, distinct from garment attributes. Folding everything into one embedding just recreates the bag-of-words problem at a higher level of abstraction. Treating garment, scene, and style as three separate, independently matchable axes, rather than one blended caption, is the design decision this whole approach is built on.

Option A: vanilla CLIP or SigLIP

Zero engineering cost, zero fashion awareness. Fails the compositional query by construction, and underperforms on contextual and style queries since a single vector under-weights secondary attributes. Useful only as the baseline number to report, never as the submission itself. This project actually measured this baseline rather than assuming it: see Section 2 for the real recall numbers.

Option B: fine-tuned fashion dense embedding

Swapping the encoder for a fashion-tuned one (Marqo-FashionCLIP or FashionSigLIP) improves recall on garment vocabulary and color naming considerably over generic CLIP, without touching the binding problem. Reasonable as a fallback or reranking signal, weak as the sole system. This project's dense half plays exactly this role: a fallback for phrasing the schema does not capture, not the primary signal.

Option C: region and detection based retrieval

Detecting each garment with a fashion object detector, embedding each region, and matching query sub-phrases to regions solves compositionality by construction. This is structurally what Pinterest's Shop the Look and Myntra's My Stylist do in production: both name a dedicated detection step as the first stage of their pipeline, before any embedding. Highest fidelity on the compositional query, but requires a trained detector, which is exactly the kind of indexing-engineering investment the assignment asks not to over-invest in for this exercise. Worth citing as the correct answer at production scale, not worth building for this submission.

Option D: structured extraction into a symbolic schema (chosen)

A per-image JSON record binds garments to slots (upper, lower, outerwear, footwear, accessory), each with a type and color, plus scene and style as separate top-level fields. At query time, a parser converts the natural language query into the same schema, and retrieval blends an exact schema match against dense similarity on a flattened caption. This solves compositionality through explicit slot binding rather than spatial detection: it is cheaper than Option C, and it stays zero-shot in principle since nothing about the schema requires a closed label set to retrain against. Full detail and the real numbers behind this choice are in Section 2.

Option E: coarse crop plus structured attributes

A middle ground between C and D: instead of a trained detector, use pose estimation or simple quadrant crops to get two or three rough garment regions, then run the same attribute-extraction step per crop instead of on the whole image. This recovers most of Option C's binding accuracy without training anything, since pose estimation models are pretrained and available off the shelf. This is the strongest available extension if there is extra time beyond this submission, and is listed as future work rather than built here, to avoid over-investing in indexing engineering the assignment explicitly deprioritizes.

The scoring design space

Independent of which extraction method populates the schema, the scoring step itself has several shapes worth naming: a symbolic-only exact filter (fast and precise, brittle to phrasing), a dense-only cosine similarity (robust to phrasing, blind to compositionality), a weighted hybrid of the two (simple, interpretable, easy to defend and to tune), a filter-then-rerank pipeline (Pinterest's actual production pattern, structurally the same idea staged instead of blended), and a cascade that falls back to dense search when the symbolic filter is too strict or too low confidence to trust. This project builds the weighted hybrid, the easiest of these to implement and explain, and treats the cascade and reranker versions as documented future work rather than building all five for a take-home.

Section 2Chosen approach: write-up

How a query is handled, end to end

NL query
  -> query parser              (extracts garments + scene + style into the schema)
  -> symbolic score             (exact slot + type + color match against the catalog)
  -> dense score                (cosine similarity, Chroma, local embedding model)
  -> score = alpha * symbolic + (1 - alpha) * dense     (alpha = 0.6)
  -> ranked results

Indexing runs the same shape offline: each image is turned into the same schema (garments per slot, scene, style), a flattened caption is built from that schema, and both the schema and a dense embedding of the caption are written to the store. The query path never touches a heavy model; it only touches the lightweight parser and an approximate nearest-neighbor lookup, which is what makes the design's scalability argument in Section 4 hold at any catalog size.

Why this beats a single pooled embedding, measured

Rather than argue this in the abstract, the actual system was measured against a dense-only baseline (alpha set to zero, a stand-in for Option A or B's single vector) on the assignment's own compositional query: a red tie and a white shirt in a formal setting, against a color-swapped decoy image, a white tie and a red shirt.

That is not a close call. A dense-only score of 0.6667 for both images is a literal tie, the bag-of-words failure demonstrated directly rather than assumed. Both approaches score five out of five on the assignment's five curated evaluation queries at top one, since the curated queries alone are not adversarial enough to break dense-only retrieval outright; the color-swapped decoy is where the difference actually shows up.

Zero-shot capability, addressed honestly

The query parser currently in the repository is a rule-based keyword spotter, a deliberate stand-in for a real LLM parser rather than the final answer: the exact prompt and schema to swap it in are already written (this is also called out as a limitation in Section 4). A closed vocabulary is not zero-shot, and a query using a garment word, color, scene, or style outside its hardcoded lists gets no structured signal at all from it. What was fixed in this pass, without needing an external API key, is what happens in exactly that case: previously, a query the parser recognized nothing in still had its dense similarity score multiplied by a fixed 0.4, discounting the one real signal available for a genuinely zero-shot query, for no reason. Ranking was unaffected either way, since every candidate was scaled by the same constant, but the reported match score understated a good match, a perfect dense hit would show as 40 percent instead of 100. The system now detects when no structured signal was recognized and reports the dense score undiluted in that case, an honest, measurable improvement to the zero-shot story that did not require waiting on a model API key.

Real feature extraction, not only ground-truth labels

The indexing side of this project initially leaned entirely on Fashionpedia's own category labels rather than running a model over pixels, a real gap against the assignment's Part A requirement ("process the raw images into a searchable format"). This has been closed: every real image is now also run through a local CLIP model (open_clip, no API key required) to produce a genuine feature-extraction pass, persisted to an on-disk vector store. This is kept separate from live query scoring for now, since the measurement above shows plain CLIP underperforming the schema approach for garment-presence queries; it exists as a real, tested, standalone Part A deliverable and a foundation to blend in later once properly weighted, not as an unvalidated change to today's ranking behavior.

Modularity

Logic and data are kept separate throughout: catalog loading, dense scoring, symbolic scoring, and query parsing are four independent modules, each independently unit-tested, none of them aware of the others' internals. The catalog itself is plain JSON, swappable without touching any scoring code. The assignment's own Part A and Part B split is honored at the module level within one application rather than as two separate top-level directories, since a real two-directory split would add friction with no behavioral difference at this scale, while keeping the indexing and retrieval concerns cleanly separated in code.

Scalability to one million images, measured

Rather than only reason about this, the actual code was timed against the real catalog.

1.91 µs
measured cost of one symbolic-match comparison against one record
~1.9 s
that cost extrapolated linearly to one million records, still the dominant scaling risk

The symbolic scoring step currently loops over every record in plain Python. At today's scale that is irrelevant, well under a hundredth of a millisecond total. Extrapolated to one million records it comes out to roughly 1.9 seconds, which is the real, concrete thing that would need to change before scaling much past a few thousand records: move garment, scene, and style matching out of a Python loop and into the vector database's own metadata filtering (Chroma already supports this), so the filtering happens natively in the database engine rather than by iterating every record in application code on every request. Storage at one million images comes out to roughly six to eight gigabytes total across image embeddings, caption embeddings, and JSON metadata, comfortably resident in memory on a single mid-size server. The dominant cost at any scale is the per-query encoding step, the parser and the embedding model, not the size of the catalog, since that cost is paid once per query rather than once per record; caching parsed queries and eventually distilling the parser into a small local model are the levers that actually reduce it, and are listed as future work in Section 4.

Section 4Future work

4a. Extending to locations and weather

The schema already treats scene as a first-class, independently filterable field, so adding location and weather is schema growth, not an architecture change. Concretely: extend the scene vocabulary with specific place types (a shopping mall, a beach, a mountain trail) alongside the existing office, street, park, and home categories, and add a parallel weather field (sunny, rainy, snowy, overcast) populated the same way scene is, either from a vision-language model call at index time or, where the dataset carries it, from EXIF or geo metadata directly. The retrieval side needs no change at all: it already blends an arbitrary number of schema fields into the symbolic score, so a query like "a raincoat on a rainy day in a city" would immediately gain a third filterable axis the moment the field exists on both sides of the schema.

4b. Improving precision

  • Hard-negative mining. Fine-tune the dense half specifically on attribute-swapped captions (the NegCLIP and ARO line of work), the same color-swapped-decoy pattern this project already used to demonstrate the failure mode, turned into training signal rather than only an evaluation case.
  • Graduated parser-confidence fallback. The all-or-nothing version is done: a query the parser recognizes nothing in now falls back to dense-only scoring automatically. The parser already computes a confidence value between recognizing nothing and recognizing everything; the remaining piece is using that graduated value to blend alpha smoothly rather than as a binary switch, so a partially-confident parse leans further toward dense search instead of trusting an uncertain symbolic filter at full weight.
  • A learned reranker. Once real query-to-relevance feedback exists, replace the fixed weighted sum with a small cross-encoder or learning-to-rank model, mirroring Pinterest's own retrieve-then-rerank production pattern rather than inventing a new one.
  • Query parser distillation. Once enough real query-to-schema pairs are logged from the LLM parser once it is wired up, distill them into a small local classifier or NER model, cutting API latency and cost at high query volume, mirroring how Pinterest moved from many specialized models to one unified, cheaper-to-serve model over time.
  • Wiring the real vision-language model. This remains the single highest-leverage next step: it fills in color, scene, and style for the real catalog images (currently honest nulls rather than guesses), and it replaces the keyword-spotting parser with something genuinely zero-shot. The prompt and schema for both are already written; only an API key is needed to turn them on.

Two things worth naming plainly

First, a zero-shot classification attempt was made without waiting on that API key, using a local CLIP model to label scene and style directly. It did not hold up under scrutiny: rewording the prompts, still reasonable wording, flipped the predicted label distribution on the exact same 40 images from one dominant scene label to a different one, and the same instability showed up on style prompts. This was measured, documented, and the unreliable labels were not shipped, rather than presenting an unvalidated result as a working feature. A real fix needs either prompt ensembling, the approach the original CLIP zero-shot recipe actually uses, dozens of templates per class rather than one, or the real vision-language model call above; not more one-off prompt tweaking.

Second, this submission includes real engineering beyond what the assignment asks to be graded on: a Docker deployment, a CI pipeline, request tracing, and frontend polish. None of it was penalized and all of it is genuinely useful, but the assignment is explicit that ML logic is what is assessed first. If there is a next increment of time to spend on this project, wiring the real vision-language model above is worth more than further infrastructure work.
